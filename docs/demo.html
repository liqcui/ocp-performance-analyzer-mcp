组件架构模式

1️⃣ MCP服务器 (FastMCP)

基于FastMCP框架的服务器，暴露分析工具供AI代理调用。支持SSE流式通信和HTTP REST API。

2️⃣ 工具/收集器 (Tools/Collectors)

专用的Prometheus指标收集器，通过PromQL查询获取集群性能数据。

3️⃣ 分析模块 (Analysis Modules)

性能分析引擎，实现瓶颈检测、阈值对比、根因分析等功能。

4️⃣ ELT管道 (Extract-Load-Transform)

数据转换管道，将原始JSON数据转换为结构化表格和HTML可视化。

5️⃣ 存储模块 (Storage Layer)

基于DuckDB的时序数据持久化，支持历史数据分析和趋势预测。

6️⃣ AI代理 (LangGraph Agents)

智能代理层，提供对话式分析、自动报告生成和数据收集服务。

⚡ 核心功能特性

🔍 实时性能监控

通过Prometheus/Thanos实时采集200+性能指标，覆盖ETCD、网络、节点、Pod等多个维度。支持自定义时间范围和采样频率。

🤖 AI驱动的智能分析

集成LangGraph和OpenAI，提供对话式性能分析。AI代理能够理解自然语言查询，自动调用相关工具，生成专业的分析报告。

📈 历史趋势分析

基于DuckDB的时序数据存储，支持长期性能趋势分析、基线对比和异常检测。

🎯 自动瓶颈检测

多维度瓶颈分析引擎，自动识别CPU、内存、磁盘I/O、网络等性能瓶颈，并提供优化建议。

📊 可视化报告生成

自动生成HTML格式的性能报告，包含执行摘要、详细指标表格、趋势图表和优化建议。

🔌 可扩展的插件架构

基于配置文件的指标管理，支持快速添加新的监控指标和分析工具，无需修改核心代码。

🎬 性能分析演示

示例1: ETCD深度性能分析

# 场景：分析ETCD集群过去1小时的性能表现

用户查询: "分析etcd过去1小时的性能，重点关注WAL fsync和backend commit延迟"

AI代理执行流程:
1. 调用 get_etcd_performance_deep_drive(duration="1h")
2. 收集以下子系统数据:
   - 常规信息 (CPU、内存、数据库大小)
   - WAL Fsync性能 (P50/P90/P99延迟)
   - Backend Commit性能 (P50/P90/P99延迟)
   - 磁盘I/O (IOPS、吞吐量、延迟)
   - 网络I/O (带宽、包速率)
   - 节点资源使用 (CPU、内存、cgroup)
   - 压缩/碎片整理统计
3. 执行瓶颈检测算法
4. 生成分析报告

📋 分析结果示例

时间范围: 2026-04-12 14:00:00 UTC ~ 2026-04-12 15:00:00 UTC

集群状态: 健康 ✓

关键发现:
  • WAL Fsync P99延迟: 8.2ms - 优秀 (目标: <10ms)
  • Backend Commit P99延迟: 32.5ms - 需关注 (目标: <25ms)
  • etcd Pod平均CPU使用率: 45% - 正常
  • 数据库空间使用: 62% - 健康
  • 磁盘I/O Wait: 15% - 瓶颈!
  • Proposal失败率: 0% - 正常

💡 优化建议

  1. Backend Commit优化:
    • 检查磁盘性能，考虑使用更快的SSD (NVMe)
    • 确认磁盘I/O调度器设置 (建议使用noop或deadline)
    • 检查是否有其他进程竞争磁盘I/O资源
  2. 磁盘I/O Wait优化:
    • 当前15%的I/O Wait表明存在磁盘瓶颈
    • 建议使用专用的高性能存储设备
    • 考虑启用磁盘缓存或使用更大的内存
  3. 持续监控:
    • 定期检查压缩/碎片整理执行情况
    • 监控数据库增长速率，提前规划容量
    • 设置告警阈值，及时发现性能退化

示例2: 网络性能分析

# 场景：排查集群网络延迟问题

用户查询: "为什么我的Pod间通信变慢了？帮我分析网络性能"

AI代理执行流程:
1. 调用 query_network_l1_metrics(duration="30m") - 检查物理层
2. 调用 query_network_io_metrics(duration="30m") - 检查吞吐量
3. 调用 query_network_socket_tcp_metrics(duration="30m") - 检查TCP连接
4. 调用 query_network_socket_softnet_metrics(duration="30m") - 检查软中断
5. 分析数据，识别异常模式
6. 生成诊断报告

📋 网络诊断结果

网络层 指标 当前值 状态
L1物理层 丢包率 2.3% 🔴 异常
L1物理层 错误包 1,234 pps 🔴 异常
TCP 重传率 1.8% 🟡 警告
TCP TIME_WAIT连接数 8,523 🟢 正常
Softnet Dropped包 5,678 🔴 异常
网络I/O 带宽利用率 78% 🟡 接近饱和

🔧 网络问题根因分析

主要问题: 物理层丢包和Softnet Dropped包表明网络接口或驱动存在问题

排查步骤:
  1. 检查网络硬件:
    • 检查网卡状态: ethtool -S eth0
    • 查看网卡错误: dmesg | grep -i eth
    • 确认网卡驱动版本和固件版本
  2. 优化Softnet参数:
    • 增加netdev_max_backlog: sysctl -w net.core.netdev_max_backlog=5000
    • 调整RPS/RFS以平衡CPU负载
    • 考虑增加中断合并时间
  3. 带宽管理:
    • 当前78%带宽使用率接近饱和
    • 考虑升级网络带宽或实施QoS策略
    • 分析流量模式，识别异常流量源
